# Runtime Data Flow: Полный поток данных в системе

> **Кодовое имя проекта:** Adventure Engine
> **Уровень документа:** Инженерный контракт
> **Статус:** Утвержден

---

## 1. Принципы потока данных

1. **Все данные проходят через Engine.**  
   Никаких обходных путей. Frontend не обращается к БД напрямую.

2. **Frontend — только отображение.**  
   Frontend не хранит состояние игры, только получает его от Engine.

3. **Realtime — только транспорт.**  
   Realtime Layer не содержит логики, только передает данные.

4. **Events — неизменяемы.**  
   После создания событие не меняется.

5. **State — единственный источник истины.**  
   Только State Manager может менять состояние сессии.

## 2. Полный runtime flow (Диаграмма)

```
┌─────────────────────────────────────────────────────────────────┐
│                         FRONTEND                               │
│  - Игрок видит задание                                         │
│  - Отправляет ответ (команду)                                 │
└──────────────────────────────┬──────────────────────────────────┘
                               │ (1. WebSocket / HTTP)
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│                      REALTIME LAYER                            │
│  - Получает команду (PLAYER_ANSWER)                            │
│  - Проверяет соединение и комнату                              │
│  - Отправляет в Event Router                                   │
│  - НЕ СОДЕРЖИТ логики                                          │
└──────────────────────────────┬──────────────────────────────────┘
                               │ (2. Команда)
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│                      EVENT ROUTER                              │
│  - Маршрутизирует событие по типу                              │
│  - Проверяет валидность команды                                │
│  - Вызывает соответствующий обработчик                         │
│  - НЕ СОДЕРЖИТ игровой логики                                  │
└──────────────────────────────┬──────────────────────────────────┘
                               │ (3. Команда)
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│                    ENGINE ORCHESTRATOR                         │
│  - Единственная точка входа в Engine                           │
│  - Получает блокировку сессии (Lock)                           │
│  - Вызывает цепочку модулей                                    │
│  - Управляет транзакциями (Commit/Rollback)                    │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│                    EVENT PROCESSOR                             │
│  - Валидирует входные данные                                   │
│  - Проверяет, что событие относится к текущему узлу           │
│  - Подготавливает данные для Rules Engine                      │
│  - НЕ проверяет условия                                        │
└──────────────────────────────┬──────────────────────────────────┘
                               │ (4. Валидированное событие)
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│                     RULES ENGINE                               │
│  - Проверяет условия (success/fail/timeout)                    │
│  - Детерминированная функция                                   │
│  - Не сохраняет состояние                                      │
│  - Возвращает TransitionType                                   │
└──────────────────────────────┬──────────────────────────────────┘
                               │ (5. Тип перехода)
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│                  TRANSITION RESOLVER                           │
│  - Находит целевой узел по типу перехода                       │
│  - Проверяет условия перехода (если есть)                      │
│  - Возвращает ID узла или null (финиш)                         │
│  - НЕ изменяет состояние                                       │
└──────────────────────────────┬──────────────────────────────────┘
                               │ (6. Следующий узел)
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│                     STATE MANAGER                              │
│  - Единственное место, где меняется состояние                  │
│  - Обновляет: currentNodeId, score, penalties                  │
│  - Сохраняет событие в Event Log (Event Sourcing)              │
│  - Сохраняет состояние в L1 (Memory), L2 (Redis), L3 (DB)     │
│  - Возвращает новое состояние                                  │
└──────────────────────────────┬──────────────────────────────────┘
                               │ (7. Новое состояние)
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│                    EVENT LOG (Event Sourcing)                  │
│  - Сохраняет событие в БД                                      │
│  - Отмечает как обработанное (Idempotency)                     │
│  - Возвращает подтверждение                                    │
└──────────────────────────────┬──────────────────────────────────┘
                               │
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│                      REALTIME LAYER                            │
│  - Получает новое состояние                                    │
│  - Генерирует события для клиента (NODE_EXIT, NODE_ENTER)     │
│  - Отправляет клиенту по WebSocket                             │
│  - НЕ СОДЕРЖИТ логики                                          │
└──────────────────────────────┬──────────────────────────────────┘
                               │ (8. События)
                               ▼
┌─────────────────────────────────────────────────────────────────┐
│                         FRONTEND                               │
│  - Получает новые события                                      │
│  - Обновляет интерфейс (задание, счет, статус)                │
│  - Показывает следующее задание                                │
│  - НЕ хранит игровое состояние                                 │
└─────────────────────────────────────────────────────────────────┘
```

## 3. Data Lifecycle (Жизненный цикл данных)

### 3.1. Где живет состояние на каждом этапе

| Этап | Данные | Хранилище | Время жизни |
| :--- | :--- | :--- | :--- |
| **1. Frontend** | Только отображение (нет состояния) | - | - |
| **2. Realtime** | Команда (временная) | Память | < 1 сек |
| **3. Event Router** | Команда (маршрутизация) | Память | < 100 мс |
| **4. Engine Orchestrator** | Команда + Lock | Память + Redis | < 5 сек |
| **5. Event Processor** | Валидированное событие | Память | < 100 мс |
| **6. Rules Engine** | TransitionType (временный) | Память | < 50 мс |
| **7. Transition Resolver** | Следующий узел (временный) | Память | < 50 мс |
| **8. State Manager** | Состояние сессии | L1: Memory, L2: Redis, L3: DB | L1: 5-15 мин, L2: 1 час, L3: постоянно |
| **9. Event Log** | Событие (неизменяемое) | PostgreSQL | Постоянно |

### 3.2. Поток сохранения состояния

```text
1. State Manager получает новое состояние
   ↓
2. Сохраняет в L1 (Memory) — мгновенно
   ↓
3. Сохраняет в L2 (Redis) — асинхронно (каждые 10 сек)
   ↓
4. Сохраняет в L3 (PostgreSQL) — асинхронно (при завершении сессии)
```

## 4. Single Source of Truth Map

| Компонент | Что является источником истины |
| :--- | :--- |
| **Game State** | Engine (State Manager) |
| **Scenario** | Database (Builder) |
| **Event Log** | Database (Event Sourcing) |
| **Session Lock** | Redis (Lock Manager) |
| **Frontend** | Получает состояние от Engine, НЕ хранит логику |
| **Realtime Layer** | НЕ хранит состояние, только передает |

## 5. Временные диаграммы

### 5.1. Последовательность событий (успешный ответ)

```
Время │ Событие
──────┼─────────────────────────────────────────────────────────
t0    │ Игрок нажимает "Отправить" (Frontend)
      │
t0+10 │ Frontend отправляет PLAYER_ANSWER через WebSocket
      │
t0+20 │ Realtime Layer получает команду
      │
t0+30 │ Event Router направляет команду в Engine
      │
t0+40 │ Engine Orchestrator получает блокировку (Lock)
      │
t0+50 │ Event Processor валидирует данные
      │
t0+60 │ Rules Engine проверяет правильность (success)
      │
t0+70 │ Transition Resolver находит следующий узел
      │
t0+80 │ State Manager обновляет состояние (score +10)
      │
t0+90 │ Event Log сохраняет событие
      │
t0+100│ Realtime Layer отправляет NODE_EXIT и NODE_ENTER
      │
t0+110│ Frontend обновляет интерфейс
      │
t0+120│ Игрок видит новое задание
      │
      │ Общее время: ~120 мс
```

### 5.2. Параллельные события (два игрока одновременно)

```
Время │ Игрок А                     │ Игрок Б
──────┼─────────────────────────────┼─────────────────────────────
t0    │ Отправляет ответ             │
t0+10 │                              │ Отправляет ответ
t0+20 │ Lock получен (сессия А)      │ Lock занят (сессия А)
t0+30 │ Обработка ответа             │ Ожидание в очереди
t0+40 │ Lock освобожден              │
t0+50 │                              │ Lock получен (сессия А)
t0+60 │                              │ Обработка ответа
t0+70 │                              │ Lock освобожден
```

## 6. Асинхронные операции

### 6.1. Фоновые задачи (не блокируют основной поток)

| Операция | Частота | Описание |
| :--- | :--- | :--- |
| **Сохранение в Redis** | Каждые 10 сек | Синхронизация L1 → L2 |
| **Сохранение в PostgreSQL** | При завершении сессии | Синхронизация L2 → L3 |
| **Отправка аналитики** | Каждое событие | Сбор метрик в фоне |
| **Обновление лидерборда** | Каждое обновление счета | Расчет рейтинга |

### 6.2. Очереди (BullMQ)

```typescript
// Очередь для фоновых задач
const queue = new Queue('background-tasks');

// Добавление задачи
await queue.add('save-session-state', {
  sessionId: 'sess-123',
  state: newState
});

// Обработчик задачи
queue.process('save-session-state', async (job) => {
  await prisma.session_states.create({
    data: {
      session_id: job.data.sessionId,
      state: job.data.state,
      created_at: new Date()
    }
  });
});
```

## 7. Обработка ошибок в потоке

### 7.1. Типы ошибок и их обработка

| Тип ошибки | Где возникает | Действие |
| :--- | :--- | :--- |
| **Валидация** | Event Processor | Отправить ошибку клиенту, остаться на том же узле |
| **Блокировка** | Orchestrator | Вернуть 409 (Session Locked), попробовать позже |
| **БД недоступна** | State Manager | Retry (3 раза), затем fallback на Redis |
| **Redis недоступна** | State Manager | Работать напрямую с PostgreSQL |
| **Engine error** | Любой модуль | Rollback состояния, освободить lock, отправить ошибку |

### 7.2. Flow обработки ошибок

```text
1. Ошибка в Engine
   ↓
2. Engine Orchestrator перехватывает ошибку
   ↓
3. Проверка: recoverable?
   ├─ Да → отправить ошибку игроку, продолжить
   └─ Нет → перейти к recovery
         ↓
4. Snapshot restore (восстановить последнее состояние)
   ↓
5. Reprocess last event (повторно обработать событие)
   ↓
6. Continue (продолжить игру)
```

## 8. Пример полного потока (от клика до UI)

### 8.1. Игрок отвечает на задание

**Шаг 1: Игрок видит задание**
```typescript
// Frontend получает NODE_ENTER
{
  "type": "NODE_ENTER",
  "payload": {
    "nodeId": "node-2",
    "question": "Найдите код на колонне",
    "hint": "Колонна у входа в парк"
  }
}

// Frontend отображает:
// 📋 Задание 2 из 5
// Найдите код на колонне
// 💡 Подсказка: колонна у входа в парк
// [  Введите код  ]
// [  Отправить  ]
```

**Шаг 2: Игрок вводит код и нажимает "Отправить"**
```typescript
// Frontend отправляет команду
{
  "type": "PLAYER_ANSWER",
  "payload": {
    "nodeId": "node-2",
    "answer": "12345",
    "answerType": "code"
  }
}
```

**Шаг 3: Engine обрабатывает ответ**
```typescript
// Engine Orchestrator
const result = await orchestrator.processEvent(sessionId, command);

// Rules Engine
const transitionType = rulesEngine.evaluate(node, event);
// → 'success'

// Transition Resolver
const nextNodeId = transitionResolver.resolve(node, 'success');
// → 'node-3'

// State Manager
const newState = stateManager.update(sessionId, {
  currentNodeId: 'node-3',
  score: 20
});
```

**Шаг 4: Engine отправляет обновление**
```typescript
// Realtime Layer отправляет события
{
  "type": "NODE_EXIT",
  "payload": {
    "nodeId": "node-2",
    "result": "success",
    "score": 20
  }
}

{
  "type": "SCORE_UPDATE",
  "payload": {
    "score": 20,
    "delta": 10,
    "source": "answer"
  }
}

{
  "type": "NODE_ENTER",
  "payload": {
    "nodeId": "node-3",
    "question": "Сфотографируйтесь у фонтана",
    "progress": { "current": 3, "total": 5 }
  }
}
```

**Шаг 5: Frontend обновляет интерфейс**
```typescript
// Frontend получает NODE_EXIT
// Показывает: ✅ Правильно! +10 очков

// Frontend получает NODE_ENTER
// Показывает новое задание:
// 📋 Задание 3 из 5
// Сфотографируйтесь у фонтана
// [  Загрузить фото  ]
// [  Отправить  ]
```

## 9. Итоговый контракт

> **Данные текут строго по одному пути: Frontend → Realtime → Engine → Realtime → Frontend.**
>
> **Никаких обходных путей.**
>
> **Engine — единственный источник истины для состояния игры.**
>
> **Все ошибки обрабатываются и не ломают систему.**
