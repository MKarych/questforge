# 38. User Journeys & Definition of Done (Полный эталон)

## 1. Общие принципы

1. **Сценарий считается пройденным**, если пользователь достиг цели без ошибок и система сохранила корректное состояние.
2. **Любое отклонение** (редирект не туда, потеря данных, белый экран, 500-я ошибка) — это **критический баг**, который блокирует релиз.
3. **Каждый маршрут** должен быть протестирован на всех поддерживаемых устройствах и браузерах.

---

## 2. Маршруты пользователей (Full User Journeys)

### 2.1. Маршрут: Бомж-игрок (неавторизованный)

1. Заходит на сайт (`/`).
2. Видит **каталог игр** (карточки с реальными играми).
3. Нажимает на **карточку игры**.
4. Видит **страницу игры** (описание, дата, город, организатор, цена).
5. Нажимает **"Участвовать"**.
6. Система редиректит его на `/auth/login` с сообщением *"Для участия необходимо войти"*.

**Ожидаемый результат:** Ни одной 404-й или 500-й ошибки. На странице игры есть все данные.

---

### 2.2. Маршрут: Игрок (зарегистрированный, не организатор)

1. Регистрируется (`/auth/register`) → попадает на `/dashboard` (или `/games`).
2. Находит игру и нажимает **"Участвовать"**.
3. Вводит **название команды** (или выбирает существующую).
4. Попадает в **Лобби (`/play/[shareLink]`)**.
5. Видит состав команды и кнопку **"Начать игру"** (или ждет старта от организатора).
6. Проходит игру (отвечает на вопросы, отправляет фото).
7. Видит **Экран финиша** (счет, время, кнопка "Поделиться").
8. Может оставить **отзыв** на странице игры.

**Ожидаемый результат:** После регистрации пользователь залогинен. Команда создается. Игра сохраняет прогресс. На финише показывается корректный счет.

---

### 2.3. Маршрут: Организатор (создание и управление)

1. Логинится.
2. Подает заявку на организатора (`/organizer/apply`).
3. Модератор одобряет заявку → роль меняется на `ORGANIZER`.
4. Переходит в `/organizer/scenarios/create`.
5. Собирает простой сценарий (СТАРТ -> ТЕКСТ -> ФИНИШ).
6. Сохраняет сценарий → он появляется в списке `/organizer/scenarios`.
7. Переходит в `/organizer/games/create`.
8. Заполняет поля (Название, Город, Дата, выбирает сценарий).
9. Создает игру → редирект на `/organizer/games/[id]`.
10. Нажимает **"Опубликовать"** (отправляет на модерацию).
11. Модератор одобряет игру.
12. Проверяет, что игра появилась в публичном каталоге `/games`.
13. Переходит по ссылке на игру и видит Лобби (как игрок).

**Ожидаемый результат:** Все переходы работают. Игра появляется в каталоге после одобрения. Сценарий сохраняется и загружается обратно.

---

### 2.4. Маршрут: Администратор / Модератор (контроль)

1. Логинится под админом.
2. Переходит в `/admin/dashboard`.
3. Видит **статистику** (количество игр, пользователей, ожидающих модерации).
4. Переходит в `/admin/games/pending`.
5. Видит игру, которую создал организатор.
6. Открывает карточку игры → видит все данные.
7. Нажимает **"Одобрить"**.
8. Проверяет, что игра исчезла из списка "Pending" и появилась в `/games`.
9. Переходит в `/admin/organizers/applications`.
10. Видит заявку от организатора.
11. Нажимает **"Одобрить"**.
12. Проверяет, что у пользователя сменилась роль на `ORGANIZER`.

**Ожидаемый результат:** Админка доступна только по роли. Модерация меняет статусы в БД и отражается на фронтенде.

## 4. Жизненный цикл игры (все статусы)

**Статусы:** `DRAFT` → `PENDING` → `APPROVED` → `PUBLISHED` → `REGISTRATION_OPEN` → `REGISTRATION_CLOSED` → `LOBBY` → `RUNNING` → `FINISHED` → `ARCHIVED`

**Запрещенные переходы:**
- ❌ `DRAFT` → `RUNNING` (нельзя запустить без публикации)
- ❌ `APPROVED` → `FINISHED` (нельзя завершить игру, которая не начиналась)
- ❌ `ARCHIVED` → `RUNNING` (нельзя перезапустить архивную игру)

**Ожидаемый результат:** Каждый переход отрабатывается корректно и сохраняется в БД.

## 5. Конструктор сценариев

**Действия:**
1. Создать блок (перетащить из палитры).
2. Перетащить блок на холсте.
3. Соединить два блока (из порта в порт).
4. Удалить блок (Delete / Backspace).
5. Копировать блок (Ctrl+C / Cmd+C).
6. Вставить блок (Ctrl+V / Cmd+V).
7. Дублировать блок (Ctrl+D / Cmd+D).
8. Изменить свойства блока (клик → панель справа).
9. Удалить соединение (клик на ребро → Delete).
10. **Undo** (Ctrl+Z / Cmd+Z).
11. **Redo** (Ctrl+Shift+Z / Cmd+Shift+Z).
12. **Сохранить** (Ctrl+S / Cmd+S).
13. **Перезагрузить страницу** и убедиться, что все блоки сохранились.

## 6. Проверка API (все методы и статусы)

| Метод | Статус | Ожидаемый результат |
| :--- | :--- | :--- |
| `GET` | 200 | Данные получены |
| `GET` | 401 | Токен не передан |
| `GET` | 403 | Недостаточно прав |
| `GET` | 404 | Ресурс не найден |
| `POST` | 201 | Ресурс создан |
| `POST` | 422 | Ошибка валидации |
| `PUT` / `PATCH` | 200 | Ресурс обновлен |
| `DELETE` | 200 | Ресурс удален |
| `DELETE` | 404 | Ресурс не найден |

## 7. Проверка базы данных (консистентность)

1. Создать игру → проверить, что запись появилась в таблице `games`.
2. Создать сценарий → проверить, что запись появилась в `scenarios`.
3. Создать команду → проверить, что запись появилась в `teams`.
4. Удалить игру → проверить, что запись **мягко удалена** (`deleted_at` заполнен).
5. Удалить команду → проверить, что удалились связанные `team_members`.

## 8. Отказоустойчивость

### 8.1. Разрыв интернета
- Отключить интернет → игра должна показать индикатор "Офлайн".
- Вернуть интернет → игра должна синхронизироваться без потери данных.

### 8.2. Обновление страницы
- Обновить страницу во время игры → состояние должно восстановиться (текущее задание, счет, таймер).

### 8.3. Закрытие браузера
- Закрыть браузер → открыть снова → перейти по ссылке → состояние восстановлено.

---

## 9. Многопользовательский режим

1. 3 игрока одновременно отправляют ответ → система обрабатывает их последовательно (блокировка сессии).
2. 2 игрока пытаются создать команду с одинаковым названием → второй получает ошибку.
3. Игрок покидает игру → команда обновляется у остальных.
4. 2 капитана пытаются стартовать игру → срабатывает блокировка.

## 11. Безопасность (Security)

- ❌ XSS → экранирование всех пользовательских данных.
- ❌ SQL Injection → использование Prisma (ORM) и параметризованных запросов.
- ❌ CSRF → использование CSRF-токенов (или SameSite-кук).
- ❌ IDOR → проверка владельца ресурса (нельзя удалить чужую игру).
- ❌ Rate Limit → ограничение на количество запросов (100/мин).
- ❌ JWT → проверка срока действия и подписи.
