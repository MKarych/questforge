## 1. Принципы MVP

### 1.1. Что такое MVP для нас

**Цель MVP:** Доказать, что организаторам удобнее проводить игры на нашей платформе, чем на Encounter или в Telegram, а игроки могут легко находить и проходить игры.

**Гипотеза:** 
1. Организатор может создать игру за 10 минут и опубликовать её в каталоге.
2. Игрок может найти игру в каталоге, зарегистрироваться и пройти её без багов.
3. Игроки оставляют отзывы, которые помогают другим выбирать игры.

**Успех:** 3-5 организаторов проводят 10+ игр на платформе в первый месяц после запуска. 50+ игроков зарегистрировались и прошли игры.

### 1.2. Что входит в MVP

✅ Публичная часть (главная, каталог, страница игры)
✅ Регистрация и авторизация пользователей
✅ Онбординг организаторов (заявка, модерация)
✅ Создание игры (название, описание, город, дата, цена)
✅ Создание линейного сценария (без ветвлений)
✅ Типы заданий: текст, код, фото
✅ Прохождение игры по ссылке
✅ Автоматическая проверка текста/кода
✅ Ручная проверка фото
✅ Подсчет очков и штрафов
✅ История ответов
✅ Отзывы на игры (звезды + текст)
✅ Базовый интерфейс (адаптивный, работает на телефоне)
✅ Модерация игр перед публикацией

### 1.3. Что НЕ входит в MVP

❌ Мобильное приложение (только веб)
❌ Маркетплейс сценариев
❌ AI-генерация
❌ Платежи и билеты (оплата офлайн/на месте)
❌ Ветвления сценариев (DAG)
❌ Чат (можно использовать внешние мессенджеры)
❌ Сложная аналитика (кроме базовой статистики)
❌ Telegram/VK боты
❌ Система достижений и уровней

## 3. Что НЕ входит в MVP (отложено на v1.0+)

| Фича | Почему не в MVP |
| :--- | :--- |
| Маркетплейс сценариев | Нужна база авторов и сценариев |
| Мобильное приложение | Веб-версия покрывает 80% случаев |
| AI-генерация | Сложно, нужно собрать данные |
| Платежи и билеты | Нужно понять, как организаторы хотят принимать деньги |
| Ветвления (DAG) | Линейный сценарий проще и покрывает 90% случаев |
| Чат | Можно использовать внешние мессенджеры на старте |
| Telegram/VK боты | Дополнительный канал, не критичный для MVP |
| Сложная аналитика | Базовая статистика (количество игр, команд) достаточна |
| Система достижений | Усложняет MVP без явной пользы |

## 5. Метрики успеха MVP

| Метрика | Цель |
| :--- | :--- |
| Количество организаторов | 5+ |
| Количество проведенных игр | 10+ |
| Количество зарегистрированных игроков | 50+ |
| Среднее время создания игры | < 15 минут |
| Среднее время прохождения игры | < 2 часа |
| Конверсия: просмотр игры → участие | > 20% |
| Доля игр с отзывами | > 80% |
| NPS (опрос организаторов) | > 50 |

## 6. Риски и их митигация

| Риск | Вероятность | Решение |
| :--- | :--- | :--- |
| **Нет времени на разработку** | Высокая | Делать по 1-2 часа в день, не торопиться |
| **Сложность с дизайном** | Средняя | Использовать готовый CSS (Tailwind) или шаблоны |
| **Организаторы не приходят** | Средняя | Пригласить лично знакомых, дать бесплатный доступ |
| **Игроки не находят игры** | Средняя | Продвигать каталог, добавить фильтры |
| **Сложность модерации** | Средняя | Начать с ручной модерации, автоматизировать позже |
| **Баги в Engine** | Высокая | Покрыть тестами > 80% |
| **Проблемы с деплоем** | Низкая | Использовать Docker + VPS (готовый шаблон) |

## 7. Итоговый принцип

> **MVP — это не "урезанная версия большого продукта".**
>
> **MVP — это самодостаточный продукт, который решает одну проблему и решает её хорошо.**
>
> **Наша проблема:** организаторам неудобно проводить игры на Encounter, а игроки не могут легко находить игры.
>
> **Наше решение:** платформа, где организатор создает игру за 10 минут, игрок находит её в каталоге, проходит и оставляет отзыв.
