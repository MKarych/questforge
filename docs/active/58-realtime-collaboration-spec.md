# 70. Realtime Collaboration Spec

> **Дата:** 27.06.2026  
> **Статус:** Утвержден  
> **Версия:** 1.0  
> **Класс:** Архитектурный контракт (10/10)  
> **Цель:** Описать все сценарии командной работы в реальном времени.

---

## 1. Архитектурные принципы
Сервер — единственный источник истины.

Клиент — только отображение.

Все изменения проходят через Engine.

Состояние синхронизируется через WebSocket.

Локальное состояние — только кэш.

text

## 2. Каналы (WebSocket)

| Канал | Описание | Пример |
|-------|----------|--------|
| `team:{teamId}` | Внутри команды | `broadcastToTeam(teamId, event)` |
| `game:{gameId}` | Все участники игры | `broadcastToGame(gameId, event)` |
| `organizer:{userId}` | Организатору | `broadcastToOrganizer(organizerId, event)` |

## 3. События синхронизации

| Событие | Кто | Описание |
|---------|-----|----------|
| `STATE_SYNC` | Engine → Клиент | Полная синхронизация состояния |
| `NODE_ASSIGNED` | Engine → Клиент | Переход на новый узел |
| `NODE_COMPLETED` | Engine → Клиент | Узел завершён |
| `ANSWER_ACCEPTED` | Engine → Клиент | Ответ принят |
| `ANSWER_REJECTED` | Engine → Клиент | Ответ отклонён |
| `SCORE_UPDATED` | Engine → Клиент | Обновление счёта |
| `INVENTORY_UPDATED` | Engine → Клиент | Обновление инвентаря |
| `TEAM_FINISHED` | Engine → Клиент | Команда завершила игру |
| `MEMBER_JOINED` | Engine → Клиент | Новый участник |
| `MEMBER_LEFT` | Engine → Клиент | Участник покинул команду |
| `PRESENCE_UPDATE` | Engine → Клиент | Статус присутствия |
| `CHAT_MESSAGE` | Клиент ↔ Engine | Сообщение в чате |

## 4. Присутствие (Presence)

### 4.1. Heartbeat (Ping/Pong)
Клиент → Сервер: PING { timestamp: Date }
Сервер → Клиент: PONG { timestamp: Date }

text

**Интервал:** 30 секунд

### 4.2. Статусы
ONLINE → последний PING < 60 секунд
IDLE → последний PING > 60 секунд
OFFLINE → последний PING > 120 секунд

text

---

## 5. Переподключение

### 5.1. Сценарий
Клиент потерял соединение

Клиент переподключается через 3 секунды

Клиент отправляет: RECONNECT { teamId, lastEventId }

Сервер возвращает: STATE_SYNC { fullState }

text

### 5.2. Правила
Клиент НИКОГДА не доверяет своему локальному состоянию.

После переподключения — всегда полная синхронизация.

Если сессия уже завершена — возвращается финальное состояние.

text

## 6. Командный чат

### 6.1. Режимы

```typescript
enum ChatMode {
  TEAM,     // только команда
  GAME,     // все участники игры
  DIRECT,   // личка с организатором
}
6.2. Правила
text
TEAM: только участники команды, организатор не видит
GAME: все участники игры (все команды + организатор)
DIRECT: личное сообщение — только организатор и конкретный игрок
7. Offline Mode
7.1. Принцип
text
Игрок может отправлять ответы офлайн.
Они сохраняются локально.
При восстановлении связи — отправляются в очередь.
7.2. Очередь

interface PendingEvent {

8. Восстановление после падения движка
8.1. Алгоритм
text
1. Все инстансы Engine падают
2. Redis пуст
3. При запуске — загрузить последний снапшот из PostgreSQL
4. Переиграть события после снапшота
5. Восстановить все активные TeamSession
6. Уведомить всех клиентов: SERVER_RESTARTED
7. Клиенты запрашивают STATE_SYNC
8.2. Источник истины
text
PostgreSQL Event Store → Источник истины
Redis                  → Кэш
Memory                 → Временный кэш (активные сессии)
9. Архитектурные правила
text
1. Сервер — единственный источник истины.
2. Клиент никогда не доверяет локальному состоянию.
3. Все события логируются.
4. Engine — единственное место, где меняется состояние.
5. Redis — кэш, PostgreSQL — источник истины.
6. При восстановлении — всегда полная синхронизация.
7. Чат команды — отдельный канал.
8. Присутствие обновляется каждые 30 секунд.
9. Офлайн-действия синхронизируются при восстановлении.
10. Конфликты решаются по принципу "первый — канонический".
