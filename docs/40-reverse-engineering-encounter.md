# Reverse Engineering Encounter: Архитектура для превосходства

> **Цель:** Не скопировать, а превзойти, построив универсальную платформу.
> **Принцип:** Любая механика Encounter должна собираться в нашем конструкторе без кода.
> **Дата:** 23.06.2026

---

## 1. Игровые механики (Примитивы)

Наш движок должен состоять из универсальных блоков-примитивов, из которых строится любое игровое действие.

| Категория | Примитивы (Блоки) | Для каких игр Encounter | Реализовано ли у нас? |
| :--- | :--- | :--- | :--- |
| **Ввод** | Текст, Код, GPS, QR, NFC, Фото, Видео, Голос, Выбор | Схватка, Точки, Кэшинг | ✅ Частично (текст, код, фото, gps, qr) |
| **Логика** | Таймер, Подсказка, Бонус, Штраф, Агент, Диалог, Предмет | Все игры | ⏳ В планах (агент, диалог, предмет) |
| **Валидация** | Организатор, Автопроверка, AI-проверка, Голосование | Фотоохота, Фотоэкстрим | ⏳ В планах (AI) |
| **Взаимодействие** | Командная работа, Соревнование, Рандомизация, Секретность | Мокрые войны, Мозговой штурм | ✅ В процессе (команды, рейтинг) |

---

## 2. Архитектура уровней (сценариев)

Наш конструктор сценариев должен поддерживать все типы прохождения.

| Тип уровня | Описание | Пример в Encounter | Статус в нашем движке |
| :--- | :--- | :--- | :--- |
| **Линейный** | Шаг за шагом (A → B → C) | Схватка | ✅ Готово |
| **Ветвящийся** | Развилка по условиям (A → (B или C)) | Есть в сценариях | ⏳ В планах (ветвления) |
| **Случайный** | Рандомный порядок заданий | Для разнообразия | ⏳ В планах |
| **Штурмовой** | Один уровень для всех (штурм) | Мозговой штурм | ⏳ В планах (квиз-блоки) |
| **Координаторский** | Центральный судья/координатор | Агентские игры | ⏳ В планах (роль агента) |
| **Конкурсный** | Голосование и оценки | Фотоохота | ⏳ В планах (голосование) |
| **Виртуальный** | Без выхода на улицу | Викторина, Квизы | ✅ Частично (choice-блоки) |

---

## 3. Пользовательские роли (Акторы)

Система должна гибко управлять разными типами участников.

| Роль | Описание | Влияние на геймплей |
| :--- | :--- | :--- |
| **Игрок** | Основной участник | Выполняет действия, получает очки |
| **Команда** | Группа игроков | Совместные действия, общий результат |
| **Организатор** | Создатель и ведущий игры | Управляет игрой, проверяет, начисляет бонусы |
| **Автор** | Создатель сценария | Зарабатывает на продажах сценариев |
| **Агент (NPC)** | Неигровой персонаж | Выдаёт коды, проверяет задания |
| **Наблюдатель** | Судья/зритель | Оценивает, комментирует |
| **Судья** | Оценивает творческие задания | Присуждает баллы в Фотоохоте |

## 4. Игровые события (Event System)

Наша Event-система должна уметь обрабатывать все эти события и больше.

| Событие | Для чего используется | Статус в движке |
| :--- | :--- | :--- |
| `PLAYER_JOINED` / `TEAM_REGISTERED` | Старт игры | ✅ Есть |
| `NODE_STARTED` / `NODE_COMPLETED` / `NODE_FAILED` | Прохождение уровней | ✅ Есть |
| `ANSWER_SUBMITTED` / `ANSWER_ACCEPTED` / `ANSWER_REJECTED` | Валидация ответов | ✅ Есть |
| `HINT_USED` | Использование подсказки | ✅ Есть |
| `TIMER_TICK` / `TIMER_END` | Управление временем | ✅ Есть |
| `SCORE_ADDED` / `PENALTY_ADDED` | Изменение очков | ⏳ В планах (система бонусов) |
| `PHOTO_UPLOADED` / `PHOTO_RATED` | Творческие конкурсы | ⏳ В планах (голосование) |
| `NPC_INTERACTION` | Диалог с агентом | ⏳ В планах |

## 5. Способы проверки (Валидация)

Мы должны поддерживать любые способы проверки ответов игроков.

| Способ проверки | Описание | Требуется для игр | Реализация |
| :--- | :--- | :--- | :--- |
| **Точное совпадение (код/текст)** | Сравнение строк | Схватка, Точки | ✅ Готово |
| **Регулярное выражение** | Гибкий поиск по шаблону | Викторины | ⏳ В планах |
| **GPS-радиус** | Проверка местоположения | Точки, Кэшинг | ✅ Готово |
| **Сканирование QR/NFC** | Физическая метка | Схватка | ✅ Частично (QR готов) |
| **Фото (органайзер)** | Проверка фото человеком | Фотоохота | ✅ Готово |
| **AI-распознавание** | Автоматическая проверка фото | Для ускорения | ⏳ В планах |
| **Голосование (игроки)** | Оценка другими участниками | Конкурсы | ⏳ В планах |
